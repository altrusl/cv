How it works

I am a Top rated (100% JSS) freelancer on a Upwork. Work on a project is not just a coding, it's rather collaboration with a client.

I'd like to describe one such experience as a use-case that outlines my approach

It is an ongoing project for German department of education - system utilizing Vue.js and CouchDB for managing curricula for high schools.

Before i entered the project the client had been developing this system for more then a year. He changed several developers, and i can't say that they were all low-paid.

The code was a mess. Each programmer tried to do something different. Even as an MVP (Minimum viable product) it can hardly be considered. There were no coding, deploying or any other standards or conventions. No SRS (Software requirements specification) of course, no any other documentation.

After clarifying the situation i discussed with the client his business needs and goals. Then we determined basic functional requirements for the system. For that i created several UML diagrams. Then we set up the appropriate data model.

After that i started refactoring the code according to the new chosen architecture. Also i started creating documentation for the project. Its purpose is to describe the project, to simplify understanding of it for both new developers and the client. Example of such documentation can be seen here I think internal documentation is a very important artifact in software development. Many programmers don't document their work because they want to bind a client to them.

After implementation of new architecture was done a development of new features went much faster. The application has become much more reliable, scalable and maintainable. I had set up CI/CD for automatic project build after each commit to the repository and did some other DevOps stuff. This saved a lot of developer time.

Sportsbook

Legal licensed online sportsbook targeted to the North American market. Applications of this kind place high demands on the security, reliability, performance and scalability of the system. My roles included architect, lead developer, DevOps engineer.

I wrote the core of the website (javascript, Vue.js, PHP, MySQL), integrated it with Sportradar - the largest provider of odds data (java, RabitMQ), and with MoneyMatrix and Bambora - payment solution providers. Deployed the system on AWS EC2 cluster, tuned load balancing and security.

From technical point of view, the most interesting and difficult part was to provide to a large number of users (tens of thousands at the same time) the proper quality of services. For this purpose custom caching system was developed.

SureBets scanner

Software system that collects in real-time odds from different online sportsbooks and runs surebets (arbitrage) and other strategies to analyze and find most valuable stakes

The system grabs events/odds data using website scrapping from several major sportsbook sites - Betfair, Pinnacle, Mozzartbet and others. Odds update interval (for some particular sport) can be less then 1 second (depends on network conditions and target website performance).

As an add-on i wrote a betting automation bot on Puppeteer (javascript) which automatically places bets on selected sportsbooks.

The system runs on Hetzner servers, with hundreds and thousand asynchronously working threads fetching and analyzing gigabytes of data every minute. Technically it was a challenge to bypass sportsbook bot defense.

To search for suitable strategies Big Data and mathematical modeling methods were used.

Vuesence

This is my open source project and NPM module

In Vue.js development the use of SFCs (single file components) has become the de-facto standard. But there is no convenient tool for documenting such components like, for example, javadoc for java classes. The existing solutions on the market (vuedoc, vuese, jsdoc, styleguidist) lack functionality.

This product is designed to fill this niche. It parses SFC component code into markdown and then uses some website generation tool for generating the documentation in human readable form.
